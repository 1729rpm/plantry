## Grouping decision (Rajat, this session — supersedes the README's literal list)

The README listed a six-group order with Tamarind→Pantry and Besan-last as examples.
Rajat's actual instruction: **limited sections, no "Other" catch-all, Pantry rendered
last.** Concretely:

- **New fixed group order:** `Proteins and Dairy · Fruit · Vegetables · Aromatics and Herbs · Pantry`.
- **"Other" is eliminated.** All 10 current "Other" rows in `data/ingredients.md` are
  fruits (Banana, the "Fruit" placeholder, Jamun, Litchi, Mango, Papaya, Peach,
  Pineapple, Plum, Pomegranate) → they all move to the new **Fruit** group.
- **Pantry is the fallback bucket and renders last.** Any ingredient without an explicit
  group falls to Pantry (was "Other"). There is no generic catch-all section any more.
- **Besan-last / ITEM_LAST override: deferred.** Besan/Tamarind were illustrative and are
  not catalog rows; no current item needs sinking, so we do not add the override mechanism
  yet (right-size — no premature abstraction). Within-group order stays A–Z. Revisit if a
  real item needs pinning.

This is a structural canonical-data + spec change (`data/ingredients.md`, the
`GroceryGroup` enum, `docs/engine.md §6`, `docs/product.md §3`). Authorized by Rajat for
this session; ships as Stream A and is **reviewed by Rajat personally** (canonical data).

### Lane / hotspot notes

- **A is fully disjoint** from B and C (engine/data/docs vs app/web). B is group-order
  agnostic (it maps over whatever groups the query returns), so A and B do not collide.
- **B and C share `index.css`** but different rule blocks. C is small → merge C first; B
  owns the index.css rebase as later-merger.
- **B must not touch `.screen__header` padding** (owned by the live
  `header-top-padding-safe-area` stream). B routes its own horizontal padding through the
  `--pt-gutter` token in longhand (DoD rule, `docs/development.md` §4) — never the README's
  literal `env(...)`-in-shorthand form (stylelint blocks it).
- **B Convex change** needs Deploy Convex success on the preview (camelCase filename is
  already fine) and a full-flow crawl + design-compare before merge.
