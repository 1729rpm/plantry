## 2026-06-14  Within-week recency + one-HP-per-meal  (#61)

Two engine rule changes from a Rajat-directed proactive run (a menu review of the 2026-06-15 generation). No Convex rows were consumed: every consumed-id list is empty. The 2026-06-15 menu showed the same HP gravy (Chicken masala gravy) winning every Mon/Wed/Fri Menu 1 slot, and a Menu 1 HP gravy main paired with a second HP dish (Chicken salad, HP-tagged) in one meal.

- Cluster A (within-week repeats): structural; fix level rule edit + engine code + tests + simulation. Added §4 step 5 "within-week recency": a dish already placed earlier in the week being generated is treated as most-recently-used and sinks below fresh alternatives for every subsequent slot's ranking. It is applied last so it dominates steps 1 to 4 (the root cause was that step 1's longest-unused signal was being re-promoted by step 3 ingredient consolidation and step 4 Preferred=Yes). Honors the existing §4 exemptions (fruit-tagged dishes, lunch carbs in {Chapati, Rice}), so Seasonal fruit on Mon/Wed/Fri and Roti repeating stay intended. Threaded the running week picks through the generateWeek loop and the swap picker via a shared `withinWeekRecencySet` helper. Rationale: the HP pool is broad (21+ active dishes), so rotation was available once within-week recency applied; no new dish or tag warranted.
- Cluster B (one HP source per meal): structural; fix level rule edit + engine code + tests. §3 Menu 1 partner selection now excludes HP-tagged dishes from the partner/accompaniment pool when the main is the HP pick (a meal carries one HP source, not two). Keyed on the `HP` tag, never on dish names, so it holds for any HP protein (chicken on chicken, paneer on paneer). Graceful fallback: if excluding HP empties the partner pool, the unfiltered pool is used so the slot still fills. Rationale: property-based fix on the existing tag; no data or tag change warranted.
- Cluster C (cuisine variety: fish/prawn/mutton): small (one genuine gap); reviewed_no_change for fish/prawn (already active in the library and surfaced by Cluster A's new variety), and deferred for mutton to a separate `data/expansion-*` content batch (zero mutton dishes exist in the catalog; net-new dishes go through the reviewed expansion path, not this rules PR). Rationale: variety is mostly already present and unblocked by Cluster A; only red meat is truly missing.

## 2026-06-14  One-HP-per-meal (all forms) + protein diversity  (#62)

Two engine rule changes, a Rajat-directed follow-up to the 2026-06-14 run after verifying the regenerated 2026-06-15 week. No Convex rows were consumed: every consumed-id list is empty. The verification surfaced two residual gaps the prior PR (#61) did not fully close: Saturday Menu 3 generated "Chicken biryani" (HP complete_meal) beside "Chicken salad" (HP accompaniment) in one meal (the one-HP rule had been scoped to Menu 1 only), and the regenerated week's HP mains were chicken + paneer x3 with fish/prawn/mutton/egg never surfacing (the dish-level within-week rule passed because the dishes were distinct, but the protein was not diverse).

- Cluster D (one HP source per meal, all menu forms): structural; fix level rule edit + engine code + tests. Generalized #61's Menu-1-only non-HP partner filter into a §3 rule applying to every meal form (breakfast pair, breakfast single, Menu 1-4): a single meal carries at most one HP-tagged dish, so once the protein main is HP the meal's non-main positions exclude HP-tagged dishes. Added a shared `excludeHpIfMealHasHp` helper in `composition.ts` (keyed on the `HP` tag, never on names) and applied it at pick time across all forms in `generateWeek.ts`. Documented thin-pool fallback: if excluding HP empties a non-main pool, the unfiltered pool fills the slot (a second HP side beats an incomplete meal). Rationale: property-based generalization of an existing tag rule; no data or tag change warranted.
- Cluster E (protein-level within-week diversity for HP mains): structural; fix level rule edit + engine code + tests + simulation. Added §4 step 6 "within-week protein diversity": when ranking an HP-main pool, a candidate whose protein family already appeared as an HP main earlier in the week sinks below fresh-protein candidates, so a fresh protein ranks up. Defined a §4.6 protein-family normalization (Chicken / Chicken Breast / Chicken Keema collapse to Chicken; the soya forms collapse to Soyabean Chunk; every other protein maps to itself). Soft with fallback: if no fresh protein is available the slot still fills; it never narrows §3 eligibility or overrides the recency exemptions, and acts only on HP-main position pools. Threaded the running week's HP-main protein families through the generateWeek loop via a shared `proteinFamiliesUsedAsHpMain` helper. The simulation harness now asserts no single protein is the HP main 3+ times in a week (it was 4x chicken in week 1 before the fix). Rationale: the live library spans many proteins, so diversity was available once the soft preference applied; no new dish or tag warranted.
